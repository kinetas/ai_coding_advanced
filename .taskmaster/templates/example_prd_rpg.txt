# Repository Planning Graph (RPG) Method - PRD 템플릿

이 템플릿은 Microsoft Research의 RPG 방법론을 사용하여 구조화되고 의존성 인식 PRD를 만드는 방법을 (AI 또는 사람에게) 가르칩니다. 핵심 통찰: WHAT(기능적)과 HOW(구조적)을 분리한 뒤, 명시적 의존성으로 연결합니다.

## 핵심 원칙

1. **이중 시맨틱**: 기능적(역량)과 구조적(코드 구성)을 따로 생각한 뒤 매핑한다
2. **명시적 의존성**: 추측하지 말고, 무엇이 무엇에 의존하는지 항상 명시한다
3. **위상적 순서**: 기반을 먼저 구축한 뒤 그 위에 레이어를 쌓는다
4. **점진적 정제**: 넓게 시작하고 반복적으로 구체화한다

## 이 템플릿 사용법

- 각 `<instruction>` 블록의 지침을 따른다
- `<example>` 블록에서 좋은 패턴과 나쁜 패턴을 확인한다
- content 섹션에 프로젝트 세부사항을 채운다
- 이를 읽는 AI는 따라가며 RPG 방법을 학습한다
- Task Master는 결과 PRD를 의존성 인식 작업으로 파싱한다

## PRD 작성에 권장하는 도구

이 템플릿으로 PRD를 **작성**할 때(파싱이 아닌) **코드 컨텍스트 인식 AI 어시스턴트**를 사용하면 가장 좋은 결과를 얻을 수 있습니다:

**이유?** AI가 모듈, 의존성, 연동 지점에 대한 좋은 아키텍처 결정을 내리려면 기존 코드베이스를 이해해야 합니다.

**권장 도구:**
- **Claude Code** (claude-code CLI) - 구조화된 추론과 큰 컨텍스트에 적합
- **Cursor/Windsurf** - 전체 코드베이스 컨텍스트가 있는 IDE 통합
- **Gemini CLI** (gemini-cli) - 대규모 코드베이스용 넓은 컨텍스트 윈도우
- **Codex/Grok CLI** - 컨텍스트 인식 강력한 코드 생성

**참고:** PRD가 작성되면 `task-master parse-prd`는 어떤 설정된 AI 모델과도 동작합니다 — PRD 텍스트 자체만 읽으면 되며, 코드베이스는 필요하지 않습니다.
</rpg-method>

<functional-decomposition>
<instruction>
이제 코드 구조가 아니라 역량(CAPABILITIES, 시스템이 무엇을 하는지)을 생각하세요.

Step 1: 상위 역량 도메인 식별
- 생각: "이 시스템이 하는 주요 일은 무엇인가?"
- 예: Data Management, Core Processing, Presentation Layer

Step 2: 각 역량에 대해 구체적 기능 나열
- explore-exploit 전략 사용:
  * Exploit: 핵심 가치에 필요한 기능은?
  * Explore: 이 도메인을 완성하는 기능은?

Step 3: 각 기능에 대해 정의:
- Description: 한 문장으로 무엇을 하는지
- Inputs: 필요한 데이터/컨텍스트
- Outputs: 생성/반환하는 것
- Behavior: 핵심 로직 또는 변환

<example type="good">
Capability: Data Validation
  Feature: Schema validation
    - Description: 정의된 schema에 대해 JSON payload 검증
    - Inputs: JSON object, schema definition
    - Outputs: Validation result (pass/fail) + error details
    - Behavior: 필드 순회, 타입 확인, 제약 조건 적용

  Feature: Business rule validation
    - Description: 도메인별 validation 규칙 적용
    - Inputs: Validated data object, rule set
    - Outputs: Boolean + 위반 규칙 목록
    - Behavior: 규칙을 순차 실행, 실패 시 short-circuit
</example>

<example type="bad">
Capability: validation.js
  (문제: 이건 FILE이지 CAPABILITY가 아님. 구조를 기능적 사고에 섞음.)

Capability: Validation
  Feature: 데이터가 좋은지 확인
  (문제: 너무 모호함. Inputs/Outputs 없음. 실행 불가.)
</example>
</instruction>

<structural-decomposition>
<instruction>
이제 코드 구성(code organization)을 생각하세요. 역량을 실제 파일/폴더 구조에 매핑하세요.

규칙:
1. 각 역량은 모듈(폴더 또는 파일)에 매핑된다
2. 역량 내 기능은 함수/클래스에 매핑된다
3. 명확한 모듈 경계 사용 — 각 모듈은 하나의 책임만 가진다
4. 각 모듈이 export하는 것(public interface)을 정의한다

목표: "무엇을 하는지"(기능적)와 "어디에 있는지"(구조적) 사이의 명확한 매핑을 만든다.

<example type="good">
Capability: Data Validation
  → Maps to: src/validation/
    ├── schema-validator.js      (Schema validation feature)
    ├── rule-validator.js         (Business rule validation feature)
    └── index.js                  (Public exports)

Exports:
  - validateSchema(data, schema)
  - validateRules(data, rules)
</example>

<example type="bad">
Capability: Data Validation
  → Maps to: src/utils.js
  (문제: "utils"는 명확한 모듈 경계가 아님. validation 로직을 어디서 찾나?)

Capability: Data Validation
  → Maps to: src/validation/everything.js
  (문제: 하나의 거대한 파일. 유지보수를 위해 기능은 별도 파일로 매핑해야 함.)
</example>
</instruction>

<task-master-integration>
# Task Master가 이 PRD를 사용하는 방법

`task-master parse-prd <file>.txt`를 실행하면 파서는:

1. **역량 추출** → 메인 작업
   - 각 `### Capability:`가 최상위 작업이 됨

2. **기능 추출** → 서브태스크
   - 각 `#### Feature:`가 해당 역량의 서브태스크가 됨

3. **의존성 파싱** → 작업 의존성
   - `Depends on: [X, Y]`가 task.dependencies = ["X", "Y"]로 설정됨

4. **단계별 정렬** → 작업 우선순위
   - Phase 0 작업 = 최고 우선순위
   - Phase N 작업 = 낮은 우선순위, 적절한 순서

5. **테스트 전략 사용** → 테스트 생성 컨텍스트
   - 구현 중 Surgical Test Generator에 테스트 시나리오 전달

**결과**: 위상적 순서로 실행 가능한 의존성 인식 작업 그래프.

## RPG 구조가 중요한 이유

전통적인 평면 PRD는 다음을 유발합니다:
- ❌ 불명확한 작업 의존성
- ❌ 임의의 작업 순서
- ❌ 순환 의존성을 늦게 발견
- ❌ 범위가 잘못된 작업

RPG 구조 PRD는 다음을 제공합니다:
- ✅ 명시적 의존성 체인
- ✅ 위상적 실행 순서
- ✅ 명확한 모듈 경계
- ✅ 구현 전 검증된 작업 그래프
